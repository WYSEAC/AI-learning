Python基础

必问考点，准备充分

QPython的GIL是什么？对多线程有什么影响？
GIL（全局解释器锁）是CPython中的互斥锁，确保同一时刻只有一个线程执行Python字节码。

影响：
• CPU密集型任务：多线程无法利用多核，需使用multiprocessing
• I/O密集型任务：多线程有效，因为I/O操作会释放GIL
• 在AI应用中，大部分是I/O密集型（API调用、数据库查询），所以多线程仍然有用

替代方案：multiprocessing、asyncio、concurrent.futures.ThreadPoolExecutor
Q装饰器的原理和使用场景？
装饰器本质是高阶函数，接收一个函数作为参数，返回一个新的增强函数。使用@语法糖。

原理： 
def timer(func):
    import time
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__} 耗时: {time.time()-start:.2f}s")
        return result
    return wrapper

@timer
def process_documents(docs):
    # 文档处理逻辑
    pass
使用场景：日志记录、性能计时、权限验证、缓存（@lru_cache）、重试机制（tenacity库）
Q生成器和迭代器的区别？
迭代器（Iterator）：实现了__iter__()和__next__()协议的对象，通过next()逐个获取元素。
生成器（Generator）：使用yield关键字创建的函数，是迭代器的子集，更简洁。

关键区别：
• 生成器是创建迭代器的简便方式
• 生成器函数在yield处暂停，下次调用从暂停处继续
• 生成器是惰性求值，节省内存（适合处理大量文档）

在RAG项目中的应用：流式生成回答、分批处理文档
Q深拷贝和浅拷贝的区别？
浅拷贝（shallow copy）：copy.copy()，创建新对象但内部元素仍是引用。修改嵌套对象会影响原对象。
深拷贝（deep copy）：copy.deepcopy()，递归复制所有嵌套对象，完全独立。

实际场景：在Agent状态管理中，传递状态时需要深拷贝避免意外修改。

Git版本控制

日常开发必备工具

QGit rebase和merge的区别？
merge（合并）：
• 创建一个新的"合并提交"（merge commit），保留两个分支的完整历史
• 历史记录呈非线性，分支图较复杂
• 安全性高，不会改写已有提交
• 适合：合并功能分支到主分支、保留完整的开发历史

rebase（变基）：
• 将当前分支的提交"移动"到目标分支的最新提交之后
• 历史记录呈线性，更整洁
• 会改写提交历史（生成新的commit hash）
• 适合：同步主分支更新到功能分支、保持干净的提交历史

项目中的实践：本地开发时用git rebase main保持功能分支与主分支同步；合并到主分支时用git merge --no-ff feature保留合并记录。

⚠️ 黄金法则：不要对已经推送到远程的公共分支执行rebase，否则会导致协作者的提交历史混乱。
Q如何解决代码冲突？
冲突产生的原因：两个分支修改了同一文件的同一位置，Git无法自动合并。

解决流程：
1. 触发合并：执行git merge feature或git rebase main
2. 查看冲突文件：git status列出冲突文件（标记为both modified）
3. 打开冲突文件：找到冲突标记
<<<<<<< HEAD
当前分支的内容
=======
被合并分支的内容
>>>>>>> feature-branch
4. 手动解决：保留正确的内容，删除冲突标记
5. 标记已解决：git add <resolved-file>
6. 完成合并：git commit（merge）或git rebase --continue（rebase）

避免冲突的技巧：频繁同步主分支、小粒度提交、明确模块分工
QGit工作流你熟悉哪种？
常见Git工作流：

1. Git Flow：
• 分支：main（生产）、develop（开发）、feature/*（功能）、release/*（发布）、hotfix/*（修复）
• 优点：流程规范，适合发布周期长的项目
• 缺点：分支较多，流程较重

2. GitHub Flow：
• 分支：main + feature/*，通过Pull Request合并
• 优点：简单高效，适合持续部署
• 缺点：不适合多版本并行

3. Trunk-Based Development：
• 所有人在main上开发，使用Feature Flag控制未完成功能
• 优点：集成频繁，冲突少
• 缺点：需要完善的CI/CD和Feature Flag系统

项目中的实践：采用简化版Git Flow——main分支保持可发布状态，feature分支开发完成后通过PR合并，使用rebase保持历史整洁。

Q什么是RESTful API？设计原则是什么？
REST（Representational State Transfer）是一种API设计风格。

核心原则：
• 资源导向：URL表示资源（名词），如/documents、/agents
• HTTP方法语义化：GET查询、POST创建、PUT更新、DELETE删除
• 无状态：每个请求包含所有必要信息
• 统一接口：一致的URL结构和响应格式
• 分层系统：客户端不需要知道直接连接的是终端还是中间代理

在项目中的实践：POST /api/documents/upload、POST /api/chat、GET /api/agents/status
Q如何处理API的认证和授权？
认证（Authentication）：验证用户身份
授权（Authorization）：验证用户权限

常用方案：
• JWT（JSON Web Token）：无状态，适合分布式系统。FastAPI中使用fastapi.security
• API Key：简单，适合内部服务调用
• OAuth 2.0：第三方授权，适合开放平台

Q数据库连接池的作用是什么？
连接池预先创建并维护一组数据库连接，避免频繁创建/销毁连接的开销。

核心优势：
• 减少连接建立时间（TCP握手+认证约50-100ms）
• 控制最大连接数，防止数据库过载
• 提高并发处理能力

在项目中的使用：使用SQLAlchemy的create_engine配置连接池：create_engine(url, pool_size=10, max_overflow=20, pool_recycle=3600)

LangChain框架

AI应用核心框架

QLangChain的核心组件有哪些？
六大核心组件：

1. LLM/Chat Models：大语言模型接口，支持OpenAI、通义千问等
2. Prompt Templates：提示词模板，支持变量注入和Few-shot
3. Chains：链式调用，将多个组件串联成工作流
4. Memory：对话记忆，管理上下文历史
5. Agents：智能体，自主决策和工具调用
6. Tools：工具集，扩展LLM能力（搜索、计算、数据库等）

补充组件：Document Loaders（文档加载）、Text Splitters（文本分割）、Vector Stores（向量存储）、Retrievers（检索器）、Output Parsers（输出解析）

• TextLoader：加载纯文本文件

QLangChain的Callback有什么作用？
Callback机制允许在LLM调用的各个阶段插入自定义逻辑。

RAG技术

检索增强生成 — 面试重点

Q请描述RAG的完整工作流程
RAG（Retrieval-Augmented Generation）工作流程分为离线阶段和在线阶段：

离线阶段（数据准备）：
1. 文档加载：使用Document Loader读取多格式文档
2. 文本分割：将长文档切分为合适大小的文本块（chunk）
3. 向量化：使用Embedding模型将文本块转为向量表示
4. 存储：将向量存入向量数据库（Chroma/Qdrant）

在线阶段（查询处理）：
1. 问题向量化：将用户问题转为向量
2. 相似度检索：在向量数据库中检索最相关的文本块
3. 上下文组装：将检索结果组装为Prompt上下文
4. LLM生成：将上下文+问题发送给LLM生成回答
5. 结果返回：将生成的回答返回给用户
Q文档分割有哪些策略？各有什么优缺点？
策略原理优点缺点
固定长度分割按字符数切分简单可控可能截断语义
递归字符分割按分隔符层级切分保持语义完整块大小不均匀
文档结构分割按标题/段落切分语义最完整依赖文档结构
语义分割基于Embedding相似度语义最优计算成本高
Token分割按Token数切分精确控制Token不考虑语义
项目选择：使用RecursiveCharacterTextSplitter，设置chunk_size=1000, chunk_overlap=200，中文分隔符优先使用"。\n\n"、"。"
Q向量数据库是如何工作的？Chroma和Qdrant有什么区别？
向量数据库工作原理：
1. 将文本通过Embedding模型转为高维向量（如1536维）
2. 建立向量索引（HNSW/IVF等算法）
3. 查询时计算向量间的相似度（余弦相似度/欧氏距离）
4. 返回Top-K最相似的向量及其关联数据

Chroma vs Qdrant：
• Chroma：轻量级，嵌入式部署，适合开发和小规模应用，API简洁
• Qdrant：高性能，支持分布式部署，过滤功能强大，适合生产环境

项目选择：开发阶段用Chroma（快速原型），生产环境考虑迁移到Qdrant
Q如何评估RAG系统的效果？有哪些指标？
检索质量指标：
• 召回率（Recall）：相关文档被检索到的比例
• 精确率（Precision）：检索到的文档中相关的比例
• MRR（平均倒数排名）：第一个相关结果的排名倒数均值
• NDCG：考虑位置权重的归一化折损累计增益

生成质量指标：
• 忠实度（Faithfulness）：回答是否基于检索上下文
• 答案相关性：回答是否切题
• 上下文利用率：检索内容被有效利用的程度

评估工具：RAGAS框架、LangSmith、自定义评估脚本
Q遇到"幻觉"问题怎么解决？
LLM幻觉是指模型生成看似合理但实际错误的内容。

解决方案：
1. Prompt约束：明确要求"只基于提供的上下文回答，不知道就说不知道"
2. 提高检索质量：优化chunk策略、增加检索数量、使用重排序
3. 引用溯源：要求模型标注信息来源（对应哪个文档片段）
4. 后处理验证：对生成结果进行事实核查
5. 温度调低：降低LLM的temperature参数（0-0.3）
6. 混合检索：结合关键词检索和语义检索

项目实践：在Prompt中加入了严格的约束条件，并将temperature设为0.1，幻觉率显著降低。

多Agent系统

智能体协作设计

QAgent的核心能力有哪些？
Agent三大核心能力：

1. 规划（Planning）：将复杂任务分解为子任务，制定执行计划
2. 记忆（Memory）：存储和检索历史信息（短期记忆+长期记忆）
3. 工具使用（Tool Use）：调用外部工具/API完成特定功能

扩展能力：
• 推理（Reasoning）：Chain-of-Thought、ReAct等推理框架
• 自我反思（Self-Reflection）：评估自身输出质量并改进
• 协作（Collaboration）：与其他Agent分工协作
Q多Agent协作有哪些模式？
常见协作模式：

1. 顺序协作（Sequential）：Agent按固定顺序传递任务（如：意图识别→信息检索→回答生成）
2. 层级协作（Hierarchical）：主Agent分配任务给子Agent，汇总结果
3. 并行协作（Parallel）：多个Agent同时处理不同子任务
4. 辩论协作（Debate）：多个Agent讨论并达成共识
5. 黑板模式（Blackboard）：共享工作空间，Agent读写共享状态

项目选择：采用顺序协作+层级协作的混合模式，意图识别Agent作为入口，根据意图路由到不同的专业Agent。
QLangGraph的主要作用是什么？
LangGraph是LangChain生态中用于构建有状态、多步骤Agent应用的框架。

核心概念：
• StateGraph：基于图的工作流引擎，节点是处理函数，边是流转条件
• State（状态）：在节点间传递的共享数据结构（TypedDict）
• Conditional Edges：基于条件的动态路由
• Checkpointing：状态持久化，支持中断和恢复

为什么选择LangGraph：
• 比LangChain的AgentExecutor更灵活可控
• 支持复杂的工作流（循环、条件分支、人工介入）
• 内置状态管理和持久化
• 可视化工作流图
Q如何设计Agent之间的通信机制？
通信机制设计要点：

1. 消息格式标准化：定义统一的Agent间消息结构（发送者、接收者、内容、类型）
2. 状态共享：通过共享State传递上下文，避免信息丢失
3. 错误处理：定义Agent失败时的降级策略
4. 超时控制：设置每个Agent的执行超时时间
5. 日志追踪：记录Agent间的消息流转，便于调试

项目实践：使用LangGraph的TypedDict作为共享State，每个Agent读取所需字段并写入结果字段，通过条件边实现动态路由。

QAgent之间的冲突是如何处理的？
冲突场景与解决方案：

1. 意图模糊冲突：用户同时涉及多个意图
   → 意图识别Agent返回多个意图及置信度，按优先级依次处理

2. 资源竞争冲突：多个Agent同时访问同一资源
   → 使用Redis分布式锁，确保操作的原子性

3. 结果不一致冲突：不同Agent返回矛盾的信息
   → 设计优先级规则，订单Agent的结果优先于产品Agent

4. 超时冲突：某个Agent执行时间过长
   → 设置超时机制（5s），超时后降级处理或转人工
Q系统如何保证状态的一致性？
状态一致性方案：

1. 共享State：使用LangGraph的TypedDict作为全局状态，所有Agent读写同一份State
2. 状态快照：利用LangGraph的Checkpointing功能，每步执行后保存状态快照
3. 数据库持久化：关键状态（订单信息、用户会话）同步写入PostgreSQL
4. 幂等设计：每个Agent的处理函数设计为幂等的，重复执行结果一致
5. 事务管理：涉及数据库写操作时使用事务，保证原子性

from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma

# 1. 文档加载
loader = PyPDFLoader("document.pdf")
documents = loader.load()
print(f"加载了 {len(documents)} 页文档")

# 2. 文本分割（核心优化点）
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,        # 每个文本块1000字符
    chunk_overlap=200,     # 重叠200字符保持语义连贯
    separators=["\n\n", "\n", ".", ".", "！", "？", "；"]  # 中文分隔符优先
)
chunks = text_splitter.split_documents(documents)
print(f"分割为 {len(chunks)} 个文本块")

# 3. 向量化并存储
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)

from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages

# 1. 定义共享状态
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]
    user_intent: str          # 用户意图
    query_result: str        # 查询结果
    order_result: str        # 订单结果
    next_agent: str          # 下一个Agent

# 2. 意图识别Agent
def intent_agent(state: AgentState):
    """分析用户意图，决定路由到哪个Agent"""
    intent = analyze_intent(state["messages"][-1].content)
    return {"user_intent": intent, "next_agent": intent}

# 3. 产品查询Agent
def product_agent(state: AgentState):
    """基于向量检索回答产品相关问题"""
    result = search_products(state["messages"][-1].content)
    return {"query_result": result, "next_agent": "responder"}

# 4. 构建工作流图
workflow = StateGraph(AgentState)
workflow.add_node("intent", intent_agent)
workflow.add_node("product", product_agent)
workflow.add_node("order", order_agent)
workflow.add_node("responder", responder_agent)

workflow.set_entry_point("intent")
workflow.add_conditional_edges("intent", route_by_intent)
workflow.add_edge("product", "responder")
workflow.add_edge("order", "responder")
workflow.add_edge("responder", END)

app = workflow.compile()

"基于我对市场的了解和个人能力评估，我的期望薪资范围是20-28K。这个范围是基于：

1. 我已完成2个完整的AI应用项目，具备从开发到部署的实战经验
2. 我系统掌握了Python、FastAPI、LangChain、RAG等核心技术栈
3. 我具备快速学习能力和解决问题的能力，能在短时间内上手新项目
4. 根据BOSS直聘/智联招聘的数据，AI应用工程师岗位的平均薪资在这个范围内

当然，薪资只是考虑因素之一，我更看重团队技术氛围和成长空间。"
⚠️ 注意事项：
  • 不要先亮底牌，让对方先出价
  • 给出范围而非具体数字，保留谈判空间
  • 考虑总包（base + 奖金 + 期权 + 福利），不只看月薪
  • 如果薪资不理想，可以谈试用期缩短、学习培训机会等
  • 不要在第一轮面试就谈薪资，等技术面试通过后再谈

🏢

互联网公司AI部门

🏭
